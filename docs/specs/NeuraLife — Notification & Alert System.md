# NeuraLife — Notification & Alert System

*The complete specification for how every alert, insight, and communication is triggered, routed, prioritised, delivered, and tracked across every user role in the NeuraLife platform.*

---

## 1. Why This Deserves Its Own Specification

Every other NeuraLife component produces data. The Notification & Alert System is what makes that data actionable. Without it:

- A student is flagged AT_RISK at 2 AM by the ML pipeline — nobody finds out until the teacher happens to open the app
- A parent's child was absent today — they find out at dinner
- A SmartPad has been offline for 7 days — the principal never notices

The alert system is the nervous system of the platform. When something important happens, the right person is informed, through the right channel, at the right time — automatically.

**Design philosophy:**

- **Signal, not noise.** Every notification must require action or deliver insight. No vanity notifications.
- **Right person.** Alerts are role-specific. Teachers do not receive parent-level summaries. Parents do not receive fleet management alerts.
- **Right time.** AT_RISK alerts are immediate. Mastery summaries are evening. Fleet reports are morning. Timing is not arbitrary.
- **Right channel.** CRITICAL alerts use FCM push + SMS. Routine updates use in-app only. Channel matches urgency.
- **Quiet hours respected.** No pushes between 10 PM and 7 AM except genuine emergencies.

## 3. Alert Classification System

Every notification generated by the platform is classified on two axes: **severity** and **category**.

### Severity Levels

| Level | Label | What it means | Delivery behaviour |
| --- | --- | --- | --- |
| S1 | **CRITICAL** | Requires immediate human action. Student safety, data integrity, or system failure. | FCM high-priority push immediately + SMS if unread after 2 hours. Bypasses quiet hours. |
| S2 | **HIGH** | Requires action today. Academic risk, attendance concern, device failure. | FCM push immediately during hours 7 AM–10 PM. In-app badge. |
| S3 | **NORMAL** | Informational with recommended action. Insights, homework, achievements. | Batched FCM push at scheduled time. In-app. |
| S4 | **LOW** | Informational only. No action required. | In-app only. No FCM push. |

### Alert Categories

| Category | Description | Primary recipients |
| --- | --- | --- |
| **ACADEMIC** | Mastery changes, AT_RISK flags, insight delivery, gap detection | Teacher, Parent, Student |
| **ATTENDANCE** | Absence, late arrival, consecutive absences | Teacher (mark), Parent (inform), Principal (report) |
| **HOMEWORK** | Assignment created, due date approaching, submitted, not submitted | Teacher, Parent, Student |
| **SOCIAL** | NeuraSphere post status, badge earned, circle invitation | Student, Parent |
| **DEVICE** | Sync status, OTA update, battery, storage, offline duration | Principal, IT Admin |
| **ADMINISTRATIVE** | Announcements, timetable changes, meeting schedules, fee alerts | All roles (targeted) |
| **SYSTEM** | API errors, pipeline failures, model update status | IT Admin, Super Admin |

## 5. Delivery Channels — Specifications

### Firebase Cloud Messaging (FCM)

Used for all push notifications to Teacher App and Parent & Student App.

```json
// FCM payload structure — all NeuraLife push notifications
{
  "to": "{fcm_registration_token}",
  "priority": "high",
  "notification": {
    "title": "Arjun — AT_RISK in Mathematics",
    "body": "Arjun's mastery in Algebra has dropped to the 18th percentile. Tap to view his profile.",
    "sound": "default"
  },
  "data": {
    "notification_id": "uuid",
    "category": "ACADEMIC",
    "severity": "S1",
    "action_type": "OPEN_STUDENT_PROFILE",
    "action_payload": "{\"neura_id\": \"NID-2025-AP-084291\"}",
    "school_id": "SCH-AP-00142",
    "deep_link": "neuralife://student/NID-2025-AP-084291/mastery"
  }
}
```

**FCM priority rules:**

- S1 alerts: `"priority": "high"` — wakes device even in doze mode
- S2 alerts: `"priority": "high"` during 7 AM–10 PM; `"priority": "normal"` outside hours
- S3/S4 alerts: `"priority": "normal"` always

**Deep link routing on tap:**

| `action_type` | Opens |
| --- | --- |
| `OPEN_STUDENT_PROFILE` | Student 360° profile screen |
| `OPEN_CLASS_MASTERY` | Class mastery heatmap |
| `OPEN_HOMEWORK` | Homework detail |
| `OPEN_FLEET` | Device fleet dashboard |
| `OPEN_INSIGHT` | Today's insight detail |
| `OPEN_NEURASPHERE` | NeuraSphere post in question |

### SMS (MSG91)

Used for S1 alerts when FCM alone is insufficient, and for parents who may not have the app installed.

```
// SMS templates (MSG91 DLT-registered)

AT_RISK alert (Telugu):
"{student_name} గారు గణితంలో మరింత శ్రద్ధ అవసరం.
NeuraLife app తెరవండి లేదా {teacher_name} టీచర్‌ని సంప్రదించండి."

AT_RISK alert (English):
"{student_name} needs attention in {subject}.
Please open the NeuraLife app or contact {teacher_name}."

Absence alert (English):
"{student_name} was marked absent today ({date}).
Contact school if this is incorrect: {school_phone}"

Parent meeting (English):
"Meeting scheduled: {date} at {time} with {teacher_name}.
Reply CONFIRM or call {school_phone}"
```

**SMS triggers (strict — not overused):**

- S1 academic: AT_RISK flagged, consecutive absences (3 days)

- Administrative: Parent meeting confirmation, fee payment reminder (v2)

**DLT compliance:** All SMS templates registered with TRAI's Distributed Ledger Technology (DLT) system before production launch. Template registration is a legal requirement for transactional SMS in India.

### In-App Notifications (All Surfaces)

The notification inbox within each app. Rendered by Supabase Realtime subscription on the `notifications` table.

```
Teacher App — Notification Inbox:
  → Bell icon with unread count badge
  → Grouped by: Today / This Week / Earlier
  → Tapping notification deep-links to relevant screen
  → Swipe to dismiss
  → "Mark all as read" option

Parent App — Notification Inbox:
  → Same structure
  → Insights always appear here even if FCM was sent
  → Parent can set language preference: English or Telugu

Web Admin Console — Notification Panel:
  → Right-side drawer
  → Grouped by: Critical / Device / Academic / Administrative
  → Export as PDF (monthly report view)
```

**Supabase Realtime subscription per client:**

```jsx
// Teacher App — subscribe to own notifications
const subscription = supabase
  .channel('teacher-notifications')
  .on('postgres_changes', {
    event: 'INSERT',
    schema: 'public',
    table: 'notifications',
    filter: `recipient_id=eq.${teacherId}`
  }, (payload) => {
    updateNotificationBadge(payload.new);
    if (payload.new.severity === 'S1') showBannerAlert(payload.new);
  })
  .subscribe();
```

## 6. Quiet Hours Policy

No FCM or SMS pushes between **10 PM and 7 AM** except:

| Exception | Reason |
| --- | --- |
| S1 AT_RISK alert (if triggered during quiet hours) | Delivered at 7:00 AM sharp — not delayed further |
| None — SMS never during quiet hours | SMS is more intrusive. AT_RISK SMS held until 7 AM even if triggered at 2 AM. |
| System failure alerts to IT Admin | Email only, no push during quiet hours |

The ML pipeline runs nightly at 2 AM. All generated insights and alerts are queued. S3/S4 notifications fire at 8 PM (if triggered before 8 PM that day) or 7 AM next morning. S1 alerts fire at 7 AM if generated during quiet hours.

## 7. Alert Deduplication & Throttling

Without deduplication, the same event can trigger multiple notifications. Rules:

| Scenario | Rule |
| --- | --- |
| AT_RISK alert for same student, same subject, same week | Suppress duplicate. Re-alert only if severity escalates or new week begins. |
| Absence alert — parent already received FCM | Do not send SMS unless FCM unread after 4 hours |
| Homework due tomorrow — student already dismissed | Do not re-send |
| Device offline 48hr alert already sent | Do not re-send 48hr alert. Only escalate to 7-day alert. |
| Class-wide pattern detected — same class, same subject, < 7 days | Suppress. Rethink weekly at pattern detection run. |
| Multiple S1 alerts for same student on same day | Bundle into one notification: "3 urgent alerts for your class — tap to view" |

**Per-user throttling:**

- Parent: maximum 4 FCM pushes per day (excluding S1 AT_RISK)
- Teacher: maximum 6 FCM pushes per day (excluding S1)
- Principal: maximum 3 FCM pushes per day (summary-level)
- SMS: maximum 2 SMS per parent per day under any circumstances

## 9. The AT_RISK Escalation Flow

This is the most critical alert sequence in the platform. Full flow:

```
Nightly ML Pipeline (2 AM):
  → Calibration Engine runs
  → Student NID-2025-AP-084291 classified: AT_RISK in MATHEMATICS
  → INSERT into student_insights:
      { severity: 'CRITICAL', insight_type: 'AT_RISK', notification_pending: TRUE }
  → INSERT into notifications:
      { recipient_type: 'TEACHER', recipient_id: class_teacher_id,
        severity: 'S1', category: 'ACADEMIC',
        title: 'Arjun — AT_RISK in Mathematics',
        body: 'Arjun's algebra mastery has dropped to the 18th percentile...',
        status: 'PENDING' }

7:00 AM (quiet hours end):
  → Notification Dispatcher picks up PENDING S1 notifications
  → Sends FCM high-priority push to Class Teacher's device
  → UPDATE notifications SET status = 'SENT', sent_at = NOW()

[If FCM not opened by 9:00 AM (2 hours later)]:
  → Check: notifications WHERE id = X AND read_at IS NULL
  → Send SMS to parent:
      "Arjun needs attention in Mathematics.
       Please open NeuraLife or contact {teacher_name}."
  → Log: SMS sent, parent notified

[If teacher opens app and views student profile]:
  → UPDATE notifications SET read_at = NOW()
  → Supabase Realtime: badge count decrements
  → Teacher logs intervention OR schedules parent meeting
  → Both actions logged in intervention_logs

[If AT_RISK student has no teacher action within 48 hours]:
  → S1 alert escalates to Principal:
      "Arjun in Class 10-A has been AT_RISK in Mathematics for 48 hours.
       No intervention logged. Please follow up with {teacher_name}."
  → Principal notified via FCM + in-app
```

## 10. Notification Dispatcher Service

The background service that processes the `notifications` table and routes to the correct channel.

### Architecture

```
notifications table (Supabase PostgreSQL)
  → Supabase Realtime triggers Dispatcher on INSERT
  → Dispatcher evaluates:
      1. severity + current time → apply quiet hours rule
      2. recipient preferences → check notification_preferences table
      3. deduplication → check recent_notifications for same event
      4. channel selection → FCM | SMS | Email | In-App
  → Route to channel:
      FCM   → Firebase Admin SDK (Node.js)
      SMS   → MSG91 API (Node.js)
      Email → Resend SDK (Node.js)
      In-App → Already in notifications table → Supabase Realtime delivers
  → UPDATE notifications SET status = 'SENT', sent_at = NOW()
  → On channel failure:
      → Retry up to 3 times (exponential backoff: 30s, 2min, 10min)
      → If all retries fail: SET status = 'FAILED', log error
      → S1 failures: alert IT Admin immediately
```

### Dispatcher Schedule

The Dispatcher runs as a continuous process (part of API Gateway). Key scheduled jobs:

| Job | Schedule | Action |
| --- | --- | --- |
| Quiet hours release | 7:00 AM daily | Process all PENDING S1/S2 held from overnight |
| Evening batch dispatch | 8:00 PM daily | Send all S3 PENDING notifications (insights, summaries, badges) |
| AT_RISK escalation check | Every 2 hours | Check S1 alerts unread > 2hr → trigger SMS |
| Teacher inaction escalation | Every 24 hours | Check AT_RISK with no intervention after 48hr → alert principal |
| FCM token refresh check | Weekly | Remove stale FCM tokens (app uninstalled devices) |

## 11. Database Schema

```sql
CREATE TABLE notifications (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  school_id       TEXT NOT NULL,
  recipient_type  TEXT NOT NULL,       -- TEACHER | PARENT | PRINCIPAL | STUDENT | IT_ADMIN
  recipient_id    UUID NOT NULL,       -- teacher.id | student.id (for parent = guardian of)
  channel         TEXT NOT NULL,       -- FCM | SMS | EMAIL | IN_APP
  category        TEXT NOT NULL,       -- ACADEMIC | ATTENDANCE | HOMEWORK | DEVICE | etc.
  severity        TEXT NOT NULL,       -- S1 | S2 | S3 | S4
  title           TEXT NOT NULL,
  body            TEXT NOT NULL,
  language        TEXT DEFAULT 'ENGLISH',
  action_type     TEXT,                -- deep link action
  action_payload  JSONB,
  deep_link       TEXT,
  related_neura_id TEXT,              -- student this notification is about (if applicable)
  status          TEXT DEFAULT 'PENDING', -- PENDING | SENT | DELIVERED | READ | FAILED
  sent_at         TIMESTAMPTZ,
  delivered_at    TIMESTAMPTZ,
  read_at         TIMESTAMPTZ,
  failure_reason  TEXT,
  retry_count     INTEGER DEFAULT 0,
  hold_until      TIMESTAMPTZ,         -- quiet hours: set to 7 AM if generated at night
  created_at      TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_notifications_recipient    ON notifications(recipient_id, status);
CREATE INDEX idx_notifications_school       ON notifications(school_id, created_at);
CREATE INDEX idx_notifications_pending      ON notifications(status, hold_until) WHERE status = 'PENDING';
CREATE INDEX idx_notifications_severity     ON notifications(severity, status);

-- ════════════════════════════════════════
-- FCM TOKENS
-- ════════════════════════════════════════

CREATE TABLE fcm_tokens (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id         UUID NOT NULL,
  user_type       TEXT NOT NULL,       -- TEACHER | PARENT | PRINCIPAL
  token           TEXT NOT NULL UNIQUE,
  device_platform TEXT DEFAULT 'ANDROID',
  app_version     TEXT,
  registered_at   TIMESTAMPTZ DEFAULT NOW(),
  last_used_at    TIMESTAMPTZ,
  is_active       BOOLEAN DEFAULT TRUE
);

-- ════════════════════════════════════════
-- NOTIFICATION PREFERENCES
-- ════════════════════════════════════════

CREATE TABLE notification_preferences (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id         UUID NOT NULL,
  user_type       TEXT NOT NULL,
  language        TEXT DEFAULT 'ENGLISH',   -- ENGLISH | TELUGU
  -- Per-category FCM preference (fixed categories cannot be changed)
  academic_fcm    BOOLEAN DEFAULT TRUE,
  homework_fcm    BOOLEAN DEFAULT TRUE,
  social_fcm      BOOLEAN DEFAULT TRUE,
  device_fcm      BOOLEAN DEFAULT TRUE,
  admin_fcm       BOOLEAN DEFAULT TRUE,
  -- SMS preference
  sms_enabled     BOOLEAN DEFAULT TRUE,
  updated_at      TIMESTAMPTZ DEFAULT NOW(),
  UNIQUE(user_id, user_type)
);

-- ════════════════════════════════════════
-- DEDUPLICATION TRACKING
-- ════════════════════════════════════════

CREATE TABLE notification_dedup (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  dedup_key       TEXT NOT NULL UNIQUE,
  -- Format: {category}:{recipient_id}:{related_entity}:{time_window}
  -- Example: AT_RISK:teacher-uuid:NID-2025-AP-084291:2025-W37
  notification_id UUID REFERENCES notifications(id),
  expires_at      TIMESTAMPTZ NOT NULL,
  created_at      TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_dedup_key ON notification_dedup(dedup_key);
CREATE INDEX idx_dedup_expires ON notification_dedup(expires_at);
```

## 15. Confirmed Decisions

| Decision | Rationale |
| --- | --- |
| FCM for push, MSG91 for SMS, Resend for email | Best-in-class for each channel. MSG91 is India's leading transactional SMS provider with DLT compliance. |
| Four severity levels (S1–S4) | Prevents notification fatigue. Users learn that S1 requires immediate action. |
| Quiet hours: 10 PM – 7 AM | Respects sleep. Reduces app uninstalls from disruptive pings. S1 alerts held to 7 AM, not indefinitely dropped. |
| AT_RISK alert: FCM → SMS if unread 2 hours | Double-channel ensures the teacher actually sees it. SMS is the fallback, not the first contact. |
| Parent max 4 FCM per day | Based on push notification fatigue research. Above 4/day, users disable notifications entirely. |
| Deduplication by key + time window | Prevents the same event from generating duplicate notifications across pipeline reruns. |
| Templates in code, not database | Copy changes require a deploy — intentional quality gate. Prevents accidental bad translations reaching parents. |
| Telugu templates by default for Telugu-medium schools | Detected from school's medium setting at onboarding. Parent can always switch to English. |
| No WhatsApp in v1 | WhatsApp Business API requires Meta approval (2–4 weeks) and per-conversation charges. Evaluate for v2 based on parent feedback. |
| AT_RISK 48-hour escalation to principal | Accountability loop. Teacher is not the last line — principal sees if interventions are not happening. |
